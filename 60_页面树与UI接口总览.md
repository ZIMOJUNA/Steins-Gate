# 页面树与UI接口总览

## 1. 这份文档是谁写的、写来干什么

这份文档当前主要由 **总策划 / PM** 负责维护，  
后续如果 UI 负责人明确，也可以转给 UI 负责人继续维护。

这份文档不是让别人替 UI、程序把所有页面细节、所有交互状态、所有界面接口一次性全写完，  
而是先把这版 Demo 的**有哪些页面、这些页面各自干什么、页面之间大概怎么跳、哪些页面是首版必须保的**这些事情说清楚。

这份文档当前主要解决这些问题：

- 这版 Demo 到底有哪些页面
- 哪些页面是主流程必须保留的
- 页面之间大概怎么流转
- 每个页面大致承担什么功能
- 哪些页面当前可以简化
- 页面没做完时怎么先保主流程
- 任务、系统、程序、UI、测试该怎么理解页面结构

一句话说：

> 这份文档不是为了先把 UI 设计得很满，而是为了先把页面骨架和主跳转链立住，让主流程在界面层也能跑起来。

---

## 2. 当前这份文档最该做的事情

当前这个阶段，这份文档最该做的不是把所有页面细节展开，  
而是先把首版 Demo 真正会用到的页面结构定下来。

### 2.1 先定“有哪些页面”
要先明确：
- 主菜单要不要保
- 探索页是不是主入口
- 对话页和终端页怎么区分
- 结果页是否独立存在
- 战斗页是否保留
- 结束页怎么收束

### 2.2 先定“哪些页面是 P0”
不是所有页面都要一开始做满。  
当前最重要的是先区分：
- 没有就跑不起来的核心页
- 可以先简化的辅助页
- 可以合并或后置的页面

### 2.3 先定“页面之间怎么跳”
要尽早说明：
- 玩家从哪里进场景
- 从场景怎么进入对话 / 终端 / 结果
- 结果页怎么回场景或进下一步
- 战斗如果保留怎么接
- 最后怎么走到结束页

### 2.4 先定“每个页面大概干什么”
不用一开始写很细，  
但至少要说清：
- 这个页面主要承载什么内容
- 它服务任务、系统还是战斗
- 它是入口页、过程页还是结果页
- 它在主流程里处于什么位置

### 2.5 先定“没做完时怎么处理”
当前很重要的一点是：
- 页面不完整时能不能占位
- 某页不做时能不能并到结果页
- 战斗页不上时能不能直接用事件结果替代
- 文本没接入时能不能先挂占位内容

---

## 3. 当前这份文档不该干的事情

当前这个阶段，最需要克制的是这些：

- 不先把所有页面都拆得很细
- 不先把 UI 风格稿和交互动效都写进来
- 不先把程序接口细案写成完整规格书
- 不为了“看起来完整”新增很多非必要页面
- 不让页面结构复杂度超过当前 Demo 体量
- 不替 UI 或程序把专业细节全写掉

一句话说：

> 先保证页面结构能承托主流程，再谈界面表现和细节深度。

---

## 4. 现在最该盯住的几个问题

### 4.1 页面有没有真的服务主流程
要一直问：
- 这个页面是不是主流程真的需要
- 没有这个页面会不会断
- 这个页面是不是可以并进别的页面
- 它是不是只是“更完整”，而不是“更必要”

### 4.2 页面会不会越拆越多
当前 Demo 体量不大，  
所以要防止这些情况：
- 一个简单结果拆成很多结果页
- 一个简单提示拆成独立系统页
- 一个终端拆成很多不必要子页
- 一个短 Demo 出现太多层页面来回切换

### 4.3 页面之间的跳转是不是清楚
不能只写“有一个终端页”“有一个对话页”，  
而要尽量说明：
- 从哪里进去
- 怎么出来
- 失败后回哪里
- 结果页如何承接下一步
- 玩家会不会在页面层迷失流程

### 4.4 页面没做完时主链会不会断
如果：
- 战斗页不上
- 终端页没做全
- 对话内容没接完
- 结果页只有占位
- 结束页还没正式文案

主流程还能不能继续？

这是当前必须回答的问题之一。

## 5. 这份文档现在建议写哪些大类

这份文档当前不需要写得很细，  
但至少要把下面这些大类补出来。

### 5.1 页面清单
写清楚：
- 当前有哪些页面
- 哪些是 P0
- 哪些是可选
- 哪些页面以后可能合并或后置

### 5.2 页面职责
写清楚：
- 每个页面主要做什么
- 是承载入口、过程还是结果
- 主要服务任务、系统、战斗中的哪一块

### 5.3 页面跳转主链
写清楚：
- 主菜单到场景
- 场景到对话 / 终端 / 结果
- 结果到下一步
- 战斗到结果
- 最后到结束页

### 5.4 页面简化与替代方案
写清楚：
- 哪些页没做完可以占位
- 哪些页可以并页
- 战斗页不上时怎么处理
- 结果反馈不完整时怎么保底

### 5.5 页面和其他模块的关系
写清楚：
- 哪些页面服务任务推进
- 哪些页面承接系统反馈
- 哪些页面和战斗有关
- 哪些页面程序要优先接
- 哪些页面 UI 当前要优先考虑识别度和可读性

## 6. 可以直接照着维护的最小结构

下面这些结构够当前阶段使用，不需要再往完整 UI 规格文档方向扩。

---

### A. 当前页面清单

| 页面 | 主要作用 | 是否 P0 | 备注 |
|---|---|---|---|
| 主菜单 | 开始、继续、退出 | 视情况 | 可简化 |
| 中央广场探索页 | 主场景探索与交互入口 | 是 | 核心页 |
| 对话页 | 对话推进、选择、检定反馈 | 是 | 核心页 |
| 蝶茧终端页 | 制度展示、权限反馈、信息读取 | 是 | 核心页 |
| 事件结果页 | 结果反馈与下一步引导 | 是 | 核心页 |
| 战斗页 | 战斗验证或替代承接 | 视情况 | 看战斗去留 |
| Demo 结束页 | 尾钩收束与退出 | 是 | 核心页 |

---

### B. 页面职责大类

| 页面 | 当前主要承载什么 | 备注 |
|---|---|---|
| 主菜单 | 进入 Demo 或继续当前进度 | 可后置简化 |
| 探索页 | 玩家接触场景、对象和世界信息 | 主入口 |
| 对话页 | 推动信息获取、选择与部分检定 | 核心文本承载页 |
| 终端页 | 承接制度感、权限感和公告信息 | 核心制度页 |
| 结果页 | 承接事件结果、失败绕路、替代反馈 | 低成本收口页 |
| 战斗页 | 承接关键冲突验证 | 风险高时可替代 |
| 结束页 | 承接 Demo 尾钩和收束 | 最终出口 |

---

### C. 页面跳转主链

当前至少要把这条链说明白：

1. 主菜单 → 中央广场探索页
2. 中央广场探索页 → 对话页 / 终端页 / 事件结果页
3. 对话页 → 探索页 / 事件结果页
4. 终端页 → 探索页 / 事件结果页
5. 事件结果页 → 探索页 / 战斗页 / 结束页
6. 战斗页 → 事件结果页
7. 结束页 → 主菜单 / 退出

> 这里先保证“页面怎么流动”是清楚的，不要求现在就把每个子状态写满。

---

### D. 页面替代与低保方案

| 情况 | 当前怎么处理 | 备注 |
|---|---|---|
| 对话文本未定 | 用剧情占位文本先挂 | 不影响流程理解 |
| 终端子页没做完 | 合并成简化信息页或限制提示 | 先保主功能 |
| 结果页数据不完整 | 用占位结果页承接 | 先保跳转 |
| 战斗页未接入 | 用事件结果页替代 | 主流程不断 |
| 结束页文案未定 | 用尾钩占位文本 | 先保收束 |

---

### E. 跨组依赖

| 页面内容 | 需要谁配合 | 需要对方给什么 | 备注 |
|---|---|---|---|
| 对话页内容 | 任务策划 / 文案组 | 节点作用、文本或占位文本 |  |
| 终端页内容 | 系统策划 / 任务策划 | 权限反馈与信息内容 |  |
| 探索页交互 | 关卡策划 / 系统策划 | 点位与交互规则 |  |
| 战斗页去留 | 战斗策划 | 战斗是否保留与结果回流 |  |
| 页面接入 | 程序 | 页面跳转与基础逻辑 |  |
| 页面表现 | UI | 可读性与基础结构 |  |
| 页面验证 | 测试 | 流转是否正确、异常是否可回退 |  |

## 7. 在社团协作里怎么推进最有效

### 7.1 先锁核心页，再补附属页
当前最重要的是先定：
- 哪几个页面没它不行
- 哪几个页面可以先简化
- 哪几个页面可以以后再拆细

### 7.2 先锁页面职责，再谈界面表现
比起一开始就追 UI 完整表现，  
更重要的是让大家先知道：
- 这个页面用来干什么
- 它在主链哪个位置
- 和哪些模块有关系

### 7.3 先锁主跳转链，再补次级状态
当前阶段最重要的是主流程怎么流转。  
先把主链写清楚，次级状态和复杂分支可以后补。

### 7.4 先准备低保方案
页面如果是高风险项，  
就不能只写“以后补”，  
而要先写：
- 页面能不能占位
- 页面能不能并掉
- 页面能不能直接转结果页
- 页面异常时回哪里

## 8. 这份文档和其他人的关系

### 和总制作人 / 总策划
总制作人 / 总策划负责：
- 定 Demo 体量与页面规模边界
- 决定哪些页面必须保、哪些页面可以砍
- 在页面结构过重时做取舍

这份文档负责：
- 把页面骨架、主跳转链和页面职责说清楚
- 不让页面结构无边界膨胀

### 和任务策划
任务策划负责：
- 提供节点推进逻辑和页面使用场景

这份文档负责：
- 把这些任务需求落到具体页面结构中

### 和系统策划
系统策划负责：
- 提供权限、检定、状态、反馈等页面依赖

这份文档负责：
- 让这些系统反馈知道落在哪些页面里

### 和关卡策划
关卡策划负责：
- 提供探索页里的对象、点位和入口关系

这份文档负责：
- 确保场景交互能顺利流向页面

### 和战斗策划
战斗策划负责：
- 决定战斗页是否保留以及如何回流

这份文档负责：
- 给战斗页留出结构位置，或准备替代路径

### 和程序
程序最需要知道：
- 页面怎么跳
- 哪些页必须先接
- 哪些页可以占位
- 出异常后回哪里

### 和 UI
UI 最需要知道：
- 当前有哪些核心页
- 哪些页优先做结构
- 哪些页重点在可读性和识别度
- 哪些页当前不需要做太深

### 和测试
测试最需要知道：
- 页面主跳转链是什么
- 哪些页面是必经
- 哪些异常需要验证
- 页面异常时该回到哪里

---

## 9. 这份文档当前最低标准

这份文档不要求现在就写成完整 UI 规格书，  
但至少要达到这些标准：

- 能看出当前有哪些核心页面
- 能看出页面主跳转链怎么走
- 能看出每个页面大概承担什么功能
- 能看出哪些页面当前可以简化或替代
- 能看出页面没做完时怎么不让主链断掉

如果这些最基本的东西都还看不出来，  
那这份页面文档就还没有真正起作用。

---

## 10. 当前一句话执行结论

这份文档当前该做的，不是先把所有页面细节和 UI 接口写得很满，  
而是先把这版 Demo 的**页面清单、核心页面职责、主跳转链、低保替代方案、跨组依赖**立起来，  
让任务、系统、程序、UI、测试都对“页面层怎么承托主流程”有同一套理解。
