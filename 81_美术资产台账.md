# 美术分工大类清单

## 1. 文档用途

本文件用于整理《创世纪》Demo 当前阶段的美术分工大类。

这份文档不做正式台账，  
只用于先把当前需要的美术方向分出来，让全组知道：

- 现在有哪些美术内容要做
- 这些内容大致分成哪几类
- 每一类通常由什么类型的人来做
- 每一类当前重点是什么
- 哪些内容现在先不做

这份文档也兼顾给老师查看，  
所以每一类都尽量带常见举例，避免只写项目内部人才看得懂的简称。

---

## 2. 当前美术分工大类

按当前 Demo 范围，美术内容先分为以下几类：

1. 场景
2. 角色
3. UI / 终端
4. 事件反馈
5. 冲突 / 战斗
6. 特效与占位

---

## 3. 各大类当前内容

## 3.1 场景

当前包括：

- 中央广场主场景
- 主地面 / 主通路
- 核心建筑轮廓
- 远景 / 中景 / 近景分层
- 标语、公告栏、旗帜、导视牌
- 可交互点位外观
- 基础封边
- 基础场景装饰

常见分工举例：

- 场景设计：负责场景整体构图、区域摆放、空间视觉方向
- 场景原画：负责主场景气氛图、场景概念图、关键区域效果图
- 场景绘制 / 场景美术：负责把场景元素真正画出来并整理可用资源
- 场景上色：负责场景色彩统一、氛围控制、明暗关系
- 场景细化：负责补充装饰、材质细节、边角完成度
- 场景拆件：负责把完整图拆成前景、中景、后景、可交互部件
- 场景整合：负责最后把资源按项目需求整理成可接入形式

给其他组的老师们看的直白理解：

- 如果把中央广场做出来，通常不是一个人从头做完，而是会分成“先设计长什么样”“再画出来”“再上色”“再拆给程序接”这几步。

当前重点：

- 先把主舞台立住
- 先保证主路径可读
- 先保证制度感能被看见

当前先不做：

- 大量次级装饰铺满
- 非主流程区域完整装修
- 额外区域扩展

---

## 3.2 角色

当前包括：

- 关键引导对象
- 功能型 NPC
- 冲突对象 / 敌对对象
- 玩家身份表现素材

常见分工举例：

- 人物设计：负责人设方向、服装、轮廓、身份感、角色识别点
- 角色原画：负责角色立绘、设定图、三视图或关键角度图
- 立绘：负责对话中出现的角色半身图 / 全身图
- 头像：负责角色小头像、头像裁切、头像适配界面
- 表情差分：负责角色不同表情版本，例如平静、生气、警惕
- 上色：负责角色上色、光影、材质和整体色彩统一
- 清稿 / 线稿：负责把草图整理成正式可上色版本

给其他组的老师们看的直白理解：

- 例如一个关键角色，通常会先有人做“人物设计”，再有人做“立绘”，再有人做“上色”，最后再做“头像和表情差分”。

当前重点：

- 先保主流程角色
- 先保关键对象识别度
- 先保能接入场景 / 对话 / 事件

当前先不做：

- 大量次级 NPC 个性化差分
- 大量服装变体
- 大量附加角色储备

---

## 3.3 UI / 终端

当前包括：

- 主菜单基础视觉
- 探索界面基础 HUD
- 对话框、名字框、选项按钮
- 检定 / 选择反馈样式
- 事件弹窗 / 结果页框体
- Demo 结束界面
- 通用按钮和控件
- 状态图标
- 蝶茧终端首页
- 公共公告页
- 权限状态页
- 限制提示页
- 事件反馈页
- 终端图标系统
- 终端底纹 / 边框 / 状态标签
- 权限区分样式

常见分工举例：

- UI 设计：负责界面排版、按钮位置、信息层级、页面样式
- UI 视觉：负责把线框和结构做成正式视觉稿
- 图标设计：负责状态图标、权限图标、锁定图标等小图形
- 终端界面设计：负责蝶茧终端这一套专门界面的风格
- 按钮 / 控件绘制：负责按钮、页签、提示框、滚动条等元素
- 界面上色 / 视觉统一：负责一整套 UI 的颜色、边框、材质风格统一

给其他组的老师们看的直白理解：

- UI 不只是“画按钮”，而是要把“界面长什么样、信息怎么排、按钮怎么点、终端和普通界面怎么区分”这一整套做出来。
- 终端放在 UI 里，是因为它本质上也是界面，只是风格更特殊。

当前重点：

- 先保统一风格
- 先保可读和可点
- 先保主流程页面能接
- 先做出终端与普通 UI 可区分的视觉
- 先保接入、限制、禁止访问三类状态

当前先不做：

- 高复杂 UI 动效
- 次级页面精修
- 不影响流程的视觉强化
- 终端次级子页面全量展开
- 高复杂终端动效
- 与主流程无关的扩展页

---

## 3.4 事件反馈

当前包括：

- 调查成功 / 失败提示
- 检定成功 / 失败提示
- 权限不足提示
- 新信息获取提示
- 新目标更新提示
- 事件触发弹窗
- 成功 / 绕路 / 替代推进结果页
- 尾钩收束页

常见分工举例：

- 结果页设计：负责成功、失败、绕路、替代推进等结果页样式
- 提示框设计：负责提示弹窗、通知框、状态提示
- 信息反馈视觉：负责“玩家做了什么，系统怎么回应”的画面表达
- 文本承载页面设计：负责结果页和尾钩页里文字的阅读排版

给其他组的老师们看的直白理解：

- 这部分可以理解成“玩家操作之后，系统拿什么画面告诉玩家发生了什么”。

当前重点：

- 先让玩家看懂发生了什么
- 先让玩家看懂当前结果
- 先让玩家知道下一步去哪里

当前先不做：

- 大量变化版式
- 非关键结果页精修
- 纯装饰性反馈强化

---

## 3.5 冲突 / 战斗

当前包括两种情况：

### 如保真实战斗
包括：

- 战斗背景
- 我方单位
- 敌方单位
- 技能图标
- 状态图标
- 目标选中反馈
- 基础命中 / 受击特效
- 战斗结算页

### 如不保真实战斗
包括：

- 冲突结果页
- 风险事件表现图
- 成败反馈样式
- 关键对象 / 敌对对象事件表现

常见分工举例：

- 战斗界面设计：负责战斗页的基本布局和视觉
- 战斗角色绘制：负责我方 / 敌方在战斗中的表现图
- 技能图标绘制：负责技能按钮和技能图标
- 战斗反馈设计：负责命中、受击、选中、结算这些反馈表现
- 战斗特效：负责命中光效、状态变化等基础视觉效果

给其他组的老师们看的直白理解：

- 如果这版 Demo 有战斗，就需要有人做“战斗场景长什么样”“敌人和我方长什么样”“技能图标和命中反馈怎么表现”。
- 如果没有真实战斗，这部分就退化成“冲突结果怎么展示”。

当前重点：

- 先保这次冲突能成立
- 先保主流程不被战斗拖死

当前先不做：

- 多场战斗资产
- 完整技能树视觉
- 高规格战斗演出

---

## 3.6 特效与占位

当前包括：

- 交互高亮
- 可调查点提示
- 权限锁定提示
- 页面切换过渡
- 基础点击反馈
- 缺失资源占位图
- 临时占位框 / 占位底图
- 默认图标 / 默认卡面

常见分工举例：

- 基础特效：负责点击反馈、交互高亮、页面切换等小型特效
- 交互提示绘制：负责调查点、锁定点、可互动点的提示样式
- 占位资源制作：负责资源没做完时先放的临时图
- 替代图标制作：负责没有正式图标时的默认版

给其他组的老师们看的直白理解：

- 这部分不是“正式美术大图”，而是为了保证联调和试玩时，资源没做完也不至于全空着。

当前重点：

- 保证联调不断
- 保证缺资源时不影响主流程
- 保证当前可区分已接入和未接入内容

当前先不做：

- 高规格氛围特效
- 大量附加演出特效
- 非关键页面的强化特效

## 5. 程序同学、策划同学和文案组同学需要向美术老师沟通的内容

这部分不是技术文档，  
而是为了减少“美术已经开始画了，但后面发现接不上、方向不对、还要返工”的情况。

简单说，就是除了告诉美术“要做什么”，  
还要把“这个东西怎么用、为什么要做、先做到什么程度就够了”讲清楚。

其中最关键的一点是：  
**尽量都要配参考图、参考页面、参考链接，或者至少给出一个明确的视觉方向。**

因为很多时候不是美术不会做，  
而是需求方脑子里有画面，但没有把那个画面真正传出去。

### 5.1 程序同学需要讲清楚的内容

程序同学和美术沟通时，建议尽量讲清楚下面这些内容：

- 这张图最后会用在哪  
  例如：是放在场景里、放在对话框里、放在按钮上，还是放在终端页里。

- 大概要做成什么尺寸和比例  
  例如：横图、竖图、头像小图、按钮小图、全屏背景，避免美术画完才发现尺寸不适合。

- 需不需要透明底  
  例如：角色立绘、图标、按钮、特效，很多都需要透明背景，不然接入会很麻烦。

- 需不需要拆层  
  例如：按钮底、按钮字、发光层分开；场景前景、中景、后景分开；终端背景和图标分开。

- 有没有不同状态  
  例如：默认、悬停、点击、禁用、锁定、已解锁，这些如果不提前说，美术可能只会交一张图。

- 这份资源是不是要动  
  例如：需不需要逐帧、切换、淡入淡出、发光、按钮反馈、翻页效果。

- 文件交付时最好是什么形式  
  例如：PNG、分层源文件、序列帧、独立图标包，或者命名要能区分版本和用途。

- 有没有现成参考图或参考页面  
  例如：直接给一个按钮参考、一个终端界面参考、一个交互反馈参考，比口头说“科技感一点”“高级一点”更容易落地。

- 哪些内容必须先交，哪些可以后补  
  例如：先给可接入版就能联调，精修和补差分可以晚一点，不要一上来就让美术把所有内容一次性做满。

给其他组的老师们看的直白理解：

- 程序同学主要要讲清楚“这张图最后怎么接进游戏里”。  
- 不是只说一句“老师帮我们画个按钮”，而是要说明这个按钮是不是要点、会不会变色、需不需要分层、最后放在哪个页面。

### 5.2 策划同学需要讲清楚的内容

策划同学和美术沟通时，建议尽量讲清楚下面这些内容：

- 这张图在流程里是干什么的  
  例如：是开场第一眼看到的主舞台，还是某次检定失败后的提示页，还是终端里的权限限制页面。

- 玩家什么时候会看到它  
  例如：一进场就看到、点终端时看到、任务推进到中段才看到、冲突后才看到。

- 它要传达什么信息  
  例如：制度感、压迫感、限制感、冷感、热闹节庆感、危险感，不然美术只知道“画”，不知道“要表达什么”。

- 它是主流程必需，还是可以后补  
  例如：中央广场主场景是必需的，但某些次级装饰、次级页面可以后补。

- 当前最低做到什么程度就能用  
  例如：先有线框、先有黑白稿、先有占位版、先有单张立绘、先不做大量差分。

- 有没有参考方向  
  例如：色调参考、材质参考、气氛参考、构图参考、类似页面参考，让美术知道靠近什么感觉。

- 有没有同类资源要统一  
  例如：同一组按钮、同一套图标、同一批终端页、同一角色差分，需要统一风格和规则。

- 有没有可直接参考的图或链接  
  例如：找相近游戏界面、终端页、角色氛围图、配色参考，让美术知道不是“自由发挥”，而是有大致目标。

- 如果赶不上，有没有替代方案  
  例如：先用占位图、先用默认头像、先用简化结果页、先不上复杂特效，避免卡死主流程。

给其他组的老师们看的直白理解：

- 策划同学主要要讲清楚“为什么要画这个、玩家会怎么看到它、它在游戏里起什么作用”。  
- 不是只说一句“这里缺一张图”，而是要把用途、情绪、优先级和最低交付标准说明白。

### 5.3 文案组同学需要讲清楚的内容

文案组同学和美术沟通时，建议尽量讲清楚下面这些内容：

- 这个角色 / 页面 / 事件在文本里是什么气质  
  例如：温和、冷淡、警惕、礼貌但疏离、制度化、神圣感、压迫感，不然美术只能凭感觉猜。

- 角色说话和表现是不是有反差  
  例如：表面很客气，但实际很强势；看起来很温和，但台词里一直在施压，这会影响立绘表情和姿态判断。

- 这一段文本的情绪重点是什么  
  例如：欢迎感、不安感、异常感、冲突感、尾钩感，方便美术判断这一页该偏“平稳”还是偏“有问题”。

- 有没有关键描述不能丢  
  例如：角色衣着里必须有某种身份标记，终端文案里必须有“权限限制”的感觉，结果页里必须突出“失败但还能继续”。

- 同一个角色 / 页面前后有没有状态变化  
  例如：前面是平静，后面变警惕；前面是开放接入，后面变限制接入，这会影响差分和页面状态设计。

- 这部分文字是正式定稿，还是临时占位  
  例如：如果只是占位文本，要提前告诉美术哪些地方后面还会改，避免按临时稿画得太死。

- 哪些内容必须跟文字一起看  
  例如：某句台词要配某种表情、某个提示页必须和某种结果文案对应，不然最后图和字可能是分开的。

- 有没有能帮助理解气质的参考图或链接  
  例如：角色气质参考、服装参考、页面氛围参考、镜头情绪参考，有时候一张参考图比半页文字更容易让美术理解。

- 如果文本还没完全定，最低先给什么  
  例如：先给关键词、角色气质、场景氛围、情绪方向、页面用途，让美术先能启动，而不是一直空等全文定稿。

给其他组的老师们看的直白理解：

- 文案组同学主要要讲清楚“这个角色或页面在文字上是什么感觉、要传达什么情绪、哪些地方后面还会改”。  
- 不是只把一段台词丢给美术，而是要说明这段文字背后的角色状态、氛围和重点信息。

### 5.4 当前最实用的沟通方式

如果想减少返工，建议程序、策划、文案在找美术之前，先把下面五件事统一好：

1. 这张资源是做什么的  
2. 这张资源先做到什么程度就够  
3. 这张资源最后怎么接  
4. 这张资源如果来不及，先拿什么替代  
5. 这张资源有没有参考图、参考链接或参考案例

这样美术老师和组员拿到需求时，通常会更容易判断：

- 先做哪部分
- 谁来做更合适
- 现在画到什么程度就能交
- 哪些细化可以后面再补
